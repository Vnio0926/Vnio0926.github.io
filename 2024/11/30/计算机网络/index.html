HTTP

3.1 HTTP常见面试题

HTTP基本概念

HTTP是超文本传输协议。HTTP是一个建立在计算机之间交流通信的规范（两个以上的参与者），以及相关的各种控制及错误处理方式（行为约定和规范）的双向协议。也就是说A与B之间进行通信时，是允许中间有中转或者接力的，只要中间人遵从HTTP协议即可。

HTTP状态码

1xx：提交信息，表示目前是协议处理的中间状态，还需要后续的操作

2xx：成功，报文已经收到并被正确处理。 常见的有 200，204，206

  • 200 OK 表示一切正常。如果是非Head请求，服务器返回的响应头会有body数据
  • 204 No Content 响应头没有body数据
  • 206 Partial Content 应用于HTTP分块下载或断点续传，表示相应返回的body不是资源的全部。也是成功的一种状态码

3xx：重定向，资源位置发生变动，需要客户端重新发送请求。 常见的有 301，302，304（301和302会自动跳转，304不会）

  • 301 Moved Permanently 表示永久重定向，说明请求的资源已经不存在了，需要改用新的URL再次访问
  • 302 Found 表示临时重定向，说明请求的资源还在，但暂时需要用另一个URL来访问
  • 304 Not Modified 不具有跳转的含义，表示资源为修改，重定向已存在缓冲文件，也称缓存重定向，也就是告诉客户端可以继续使用缓存资源，用于缓存控制

4xx：客户端错误，请求报文有误，服务器无法处理。常见的有 400，403，404

  • 400 Bad Request 表示客户端请求的报文有错误，但只是个笼统的错误
  • 403 Forbidden 表示服务器禁止访问资源，并不是客户端的请求错误
  • 404 Not Found 表示请求的资源在服务器上不存在或未找到。

5xx：服务端错误，服务器在处理请求时内部发送错误。常见的有 500，501，502，503

  • 500 Internal Server Error 与400类型一样十个笼统的错误码。
  • 501 Not Implemented 表示客户端请求的功能还不支持
  • 502 Bad Gateway 通常是服务器作为网关或者代理时返回的错误码，表示服务器自身工作正常，访问后端服务器发生错误
  • 503 Service Unavailable 表示服务器很忙，暂时无法响应客户端。

HTTP常见字段

Host：客户端发送请求时，用于指定服务器的域名

Content-Length：服务器返回数据时所带的字段，表明本次回应的数据长度

Connection：常用于客户端要求服务器去使用HTTP长连接机制

HTTP长连接的特点是 只要任意一端没有提出断开连接，则保持TCP连接状态

注：TCP Keepalive和 HTTP Keep-Alive不是一个东西

Content-Type：字段用于服务器回应时，告诉客户端本次数据的格式

Content-Encoding字段说明数据的压缩方法。表示服务器返回的数据使用了什么压缩格式

Accept-Encoding：客户端在请求的时候，用该字段说明自己接收哪些压缩方法

GET与POST

GET的语义是从服务器获取指定的资源

POST的语义是根据请求负荷（报文body）对指定的资源做出处理

GET与POST方法都是安全和幂等的吗

安全和幂等的概念：

  • 在HTTP协议里，所谓的安全是指请求方法不会破坏服务器上的资源
  • 幂等的意思是多次执行相同的操作，结果都是相同的

GET方法是安全且幂等的，因为它不对服务器资源进行更改，处理。它是只读操作。所以可以对GET请求的数据做缓存，这个缓存可以做到浏览器本身上（彻底避免浏览器发请求），也可以做到代理上（如nginx），而且在浏览器中GET请求可以保存为书签。

POST是不安全且不幂等的，因为它会修改服务器上的资源。因此浏览器一般不会缓存POST请求，也不能把POST请求保存为书签

HTTP缓存技术

HTTP缓存实现方式

对于具有重复性的HTTP请求，如每次请求得到的数据都一样，可以把这对请求-响应的数据缓存在本地。HTTP缓存有两种实现方式，强制缓存和协商缓存

强制缓存

强制缓存指的是只要浏览器判断缓存没过期，则直接使用浏览器的本地缓存，决定是否使用缓存的主动性在于浏览器这边

在Status Code:200后面显示 from disk cache 则表明使用强制缓存

强制缓存是利用两个HTTP响应头部字段实现的，它们都表示资源在客户端缓存的有效期：

  • Cache-Control：是一个相对时间
  • Expires：是一个绝对时间

若HTTP响应头部同时有Cache-Control和Expires字段，则Cache-Cache的优先级高于Expires

Cache-Control选项更多，设置更精细，建议使用Cache-Control来实现强制缓存，具体实现：

  • 当浏览器第一次请求访问服务器资源时，服务器会在返回这个资源的同时，在Response头部加上Cache-COntrol，并在其中设置过期时间大小
  • 浏览器再次请求访问服务器中的该资源时，会先通过请求资源的时间与Cache-Control中设置的过期时间大小，来计算出该资源是否过期，如果没有，则使用该缓存，否则重新请求服务器
  • 服务器再次收到请求后，会再次更新Response头部的Cache-Control

协商缓存

协商缓存就是与服务端协商之后，通过协商结果来判断是否使用本地缓存。 比如状态码304就是协商缓存的一种表现形式

协商缓存可以基于两种头部实现：

第一种：请求头部中的 If-Modified-Since字段与响应头部中的 Last-Modified字段实现，这两个字段的意思是：

  • 响应头部中的 Last-Modified：表示这个响应资源的最后修改时间
  • 请求头部中的 If-Modified-Since：当资源过期了，发现响应头中具有 Last-Modified声明，则再次发起请求的时候带上Last-Modified的时间，服务器收到请求后发现有If-Modified-Since则于被请求资源的最后修改时间进行对比（Last-Modified）, 如果最后修改时间较新（大），说明资源被修改过，则返回最新资源，HTTP 200 OK；反之说明资源无新修改，响应HTTP304走缓存

第二种：请求头部中的If-None-Match字段与响应头部中的ETag字段，这两个字段的意思是

  • 响应头部中ETag：唯一标识响应资源
  • 请求头部中的If-None-Match字段：当资源过期时，浏览器发现响应头里有ETag，则再次向服务器发起请求，会将请求头If-None-Match值设置为ETag的值。服务器收到请求后进行对比，如果资源没有变化返回304，如果资源发生变化返回200

总的来说，第一种是基于时间实现的，第二种方式是基于一个唯一标识实现的。相对来说后者可以更准确的判断文件内容是否被修改，避免由于时间篡改导致的不可靠问题

注意：协商缓存这两个字段都需要配合强制缓存中的Cache-Control字段来使用，只有在未能命中强制缓存的时候，才能发起带有协商缓存字段的请求。

ETag的值是根据当前请求的资源生成的

HTTP特性

HTTP/1.1 简单，灵活，易于扩展，应用广泛和跨平台

  • HTTPS就是在HTTP（应用层）和TCP层之间增加了SSL/TLS安全传输层

  • HTTP/1.1和HTTP/2.0 传输协议使用的是TCP协议，而HTTP/3.9传输协议改成了UDP协议

HTTP/1.1缺点

HTTP协议有个优缺一体的双刃剑：分状态、明文传输，以及一大缺点 不安全

无状态的好处：不会记忆HTTP的状态，因此不会需要额外的资源记录状态信息，减轻服务器负担

无状态坏处：完成关联性操作的时候很麻烦

对于无状态的解决方法，比较简单的方式是Cookie技术

Cookie通过在请求和响应报文中写入Cookie信息来控制客户端的状态，相当于，在客户端第一次请求后，服务器会下发一个装有客户信息的小贴纸，后续客户端请求服务器的时候，带上小贴纸，服务器就能认得了了

HTTP最严重的缺点就是不安全。

HTTP/1.1的性能

HTTP协议基于TCP/IP，并且使用请求-应答的通信模式，它们的性能和这两点相关

  1. 长连接，也叫持久连接。只要任意一端没有明确提出断开连接，则保持TCP连接状态
  2. 管道（pipeline）网络传输。在同一个TCP连接里，客户端可以发起多个请求，只要第一个请求发出去了，不必等待其回来，就可以发第二个请求出去，减少整体响应时间。但是服务器必须按照接受请求的顺序发送对这些管道话请求的响应，即必须按照序列号依次对客户端发出的请求进行回应。 因此有可能会有这个场景： 假设服务端在处理A请求时耗时比较长，后续的请求就会被阻塞住，这被称为队头阻塞。所以HTTP/1.1管道解决了请求的队头阻塞，但没有解决响应的队头阻塞。 事实上，HTTP/1.1的管道化技术不是默认开启的。这个技术不常用
  3. 队头阻塞

HTTP和HTTPS

它们的区别是什么

  • HTTPS解决了HTTP不安全的缺陷，在TCP和HTTP层之间加入了TLS/SSL安全协议，报文可以加密传输
  • HTTPS在TCP三次握手之后，还需要进行SSL/TLS的握手过程，才可以加密报文传输
  • HTTP默认端口号为80，HTTPS默认端口号为443
  • HTTPS协议需要向CA（证书权威机构）申请数字证书来保证服务器的身份是可信的

HTTPS解决了HTTP的哪些问题

​ HTTP的问题：1. 窃听风险 2. 篡改风险 3. 冒充风险

​ HTTPS解决方法：1. 信息加密 （通过混合加密的方式实现信息的机密性）

​ 2. 校验机制 （通过摘要算法的方式实现完整性） 3. 身份证书（将服务器公钥放到数字证书中）

混合加密：HTTPS采用非对称加密+对称加密结合的方式

  • 在通信建立前采用非对称加密的方式交换会话密钥，后续就不再使用非对称加密
  • 在通信过程中全部使用对称加密的会话密钥的方式加密明文数据

原因： 对称加密只使用一个密钥，运算速度快，密钥必须保密，无法做到安全的密钥交换

非对称加密使用两个密钥，公钥和私钥，公钥可以任意分发而私钥保密，解决了密钥交换问题但速度慢

摘要算法+数字签名

简言之就是对内容计算出一个指纹，然后同内容一起传输给对方。然后接收方同样针对内容计算指纹，进行比对，判断是否被篡改

即 用摘要算法（哈希函数）来计算内容的哈希值，即指纹，这个哈希值是唯一的，无法逆推导

但有个新的问题：可以保证内容不被篡改，但无法保证内容+哈希是否被别人替换，因为缺少对客户端收到消息是否来源于服务端的证明

此时公钥和私钥的可以解决这个问题，不同的加密顺序带来不同的目的

公钥加密，私钥解密：能够保证内容传输的安全

私钥加密，公钥解密：能够保证消息不会被冒充。这个方法也就是数字签名方法

解决完内容信息完整（通过哈希算法），信息来源的可靠性（通过数字签名方法），最后一步还需要身份验证环节（通过CA数字证书认证机构进行数字证书的颁发）解决冒充风险

HTTPS如何建立连接，期间交互了什么

SSL/TLS协议流程：1. 客户端向服务端索要并验证公钥 2. 双方协商生产会话密钥 3. 通过密钥进行加密通信

前两步是TLS握手阶段，涉及4次通信。密钥交换的算法有两种：RSA算法和ECDHE算法

HTTPS的应用数据如何保证完整性的

TLS在实现上分为握手协议和记录协议两层。握手协议负责加密，保护应用程序数据的安全。记录协议是负责保护应用程序并验证其完整性
